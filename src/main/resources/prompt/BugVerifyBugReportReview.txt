## Common Critical Errors to Check (High Priority)

### 1. **Ignored Default Values/Initial State**
- Did the report assume a variable/field has a certain value without checking initialization?
- Example flaw: "The method returned null, violating the spec" → but the spec may guarantee non-null only after explicit initialization

### 2. **Spec Confusion Across Languages/Standards**
- Is the cited specification actually for Java/JDK, or is it from C++/POSIX/HTTP/general CS concepts?
- Example flaw: "POSIX requires..." → but Java's contract may differ from POSIX
- Verify: Does the cited javadoc/JLS section actually apply to this Java class/method?

### 3. **Overload/Override Spec Mismatch**
- Is the report citing documentation for a different method signature?
- Example flaw: Citing `List.add(E)` spec when the code calls `List.add(int, E)`
- Example flaw: Citing `Object.equals()` contract when reviewing `String.equals()` (which has additional guarantees)
- **Required check:** Match parameter types, parameter count, and declaring class precisely

### 4. **"Should" vs "Must" in Specifications**
- Does the spec use recommendation language ("should", "recommended", "typically") or mandatory language ("must", "shall", "required")?
- Example flaw: "Spec says implementations *should* throw NPE" → this is a suggestion, not a contract violation if NPE isn't thrown
- **Only "must"/"shall"/"required" statements constitute enforceable contracts**

## Systematic Review Process

### Step 1: Evidence Verification

For each major claim in the report, ask:
1. **Source Match:** Does the quoted spec precisely match the invoked method (class, signature, parameter types)?
2. **Language Precision:** Is the spec clause mandatory ("must") or advisory ("should"/"may")?
3. **Completeness:** Does the report address ALL relevant output lines, or cherry-pick supporting ones?
4. **Context:** Are initialization, preconditions, and prior state properly considered?

**Output format:**
```
CLAIM: [Quote from report]
EVIDENCE: [What supports/refutes it]
ASSESSMENT: ✓ Valid / ✗ Invalid / ? Uncertain
```

### Step 2: Alternative Explanations

For the observed failure, consider:
- **Test error scenarios:** Wrong assertion, unmet precondition, environment assumption, initialization missing
- **Spec-compliant scenarios:** Behavior is allowed/unspecified, "should" not "must", platform-specific
- **Simplest explanation:** What requires fewer assumptions?

**If a simpler explanation exists that doesn't require a JDK bug, bias toward it.**

### Step 3: Decision Framework

**For JDK_BUG verdict to stand, ALL must be true:**
- [ ] Cited spec uses mandatory language ("must"/"shall"/"required")
- [ ] Spec precisely matches the invoked method signature and class
- [ ] Spec is from official Java documentation (not C++/POSIX/etc.)
- [ ] Observed behavior directly contradicts the mandatory requirement
- [ ] Alternative explanations (test error, unmet preconditions, initialization) are ruled out

**If ANY checkbox fails, the verdict should be TESTCASE_ERROR or UNCERTAIN.**

### Step 4: Honest Assessment

Rate your confidence in the current `bug_type` conclusion:
- **High (8-10/10):** Strong evidence, critical checks passed, alternatives ruled out
- **Medium (5-7/10):** Reasonable evidence but gaps exist, some alternatives not fully ruled out
- **Low (1-4/10):** Weak evidence, critical errors found, or simpler explanations available

**If confidence is Low, you MUST recommend changing the conclusion or marking it UNCERTAIN.**

## Guiding Principles

1. **Honesty over consistency:** If evidence contradicts the original conclusion, say so clearly
2. **Precision over persuasion:** Vague claims like "seems to violate" must become "violates [spec section X] which states [quote]" or be removed
3. **Simplicity over complexity:** The explanation requiring fewest assumptions is usually correct
4. **Evidence over intuition:** "This feels wrong" is not sufficient; cite exact spec text
5. **Uncertainty is acceptable:** "Evidence is insufficient to determine bug_type" is a valid conclusion

**Remember:** Your reputation depends on accuracy, not on defending a position. Be ruthlessly honest about what the evidence actually shows.
